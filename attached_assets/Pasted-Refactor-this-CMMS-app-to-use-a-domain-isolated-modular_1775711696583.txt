Refactor this CMMS app to use a domain-isolated modular architecture. 
This is a structural refactor only — do not change any existing functionality, 
UI, or business logic. The app must work exactly the same after the refactor 
as it does before. Complete all subtasks in order.

---

SUBTASK 1 — LAZY LOAD ALL ROUTES IN App.tsx

- Convert every static page import at the top of App.tsx to use React.lazy()
- Wrap the entire <Routes> block in a <Suspense> component with a 
  fallback={<div className="flex items-center justify-center h-screen">
  <Spinner /></div>} (reuse whatever loading/spinner component already exists)
- Do not change any route paths, layouts, or navigation logic
- Example pattern to follow:
    BEFORE: import TaskDetail from './pages/TaskDetail'
    AFTER:  const TaskDetail = lazy(() => import('./pages/TaskDetail'))

SUBTASK 2 — ADD PER-DOMAIN ERROR BOUNDARIES

- Create a reusable DomainErrorBoundary component at 
  client/src/components/DomainErrorBoundary.tsx
- It must accept two props: domain (string) and children
- When a child crashes, it should display a contained error card (not 
  full screen) showing: the domain name, "Something went wrong in this 
  section", and a "Reload" button that calls window.location.reload()
- Style it to match the existing app (use existing Tailwind classes and 
  color palette)
- In App.tsx, wrap each domain's route group with its own 
  DomainErrorBoundary. Group routes by these domains:
    - Work Orders (all /tasks and /work* routes)
    - Vehicles (all /vehicle* routes)
    - Inventory (all /inventor* routes)
    - Facilities (all /facilit* and /building* routes)
    - Analytics (all /analytic* and /report* routes)
    - Messaging (all /message* and /chat* routes)
    - Admin / Settings (all /settings*, /admin*, /user* routes)
    - Everything else gets wrapped in a General boundary
- Keep the existing top-level error boundary as a last-resort fallback

SUBTASK 5 — DECOMPOSE GIANT PAGE FILES

- Identify any page component file over 1,000 lines
- For each one:
    1. Create a subfolder named after that page 
       (e.g. pages/TaskDetail/ or domains/work-orders/TaskDetail/)
    2. Move the existing file to index.tsx inside that folder
    3. Extract logical sections into focused child components in that folder:
       - Each extracted component should be under 300 lines
       - Extract data fetching and state into a custom hook (useXxx.ts) 
         in the same folder
       - The index.tsx should become an orchestrator under ~200 lines 
         that composes the child components
    4. Do not change any prop interfaces, exported types, or the 
       component's external API
- Priority order: TaskDetail.tsx first, then Work.tsx, then any others
